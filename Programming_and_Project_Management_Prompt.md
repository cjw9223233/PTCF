**專屬「程式設計 × 專案管理」高階 Prompt 模板**。

這個版本會針對你常做的事情優化：

* 專案解析
* 架構設計
* 程式碼審查
* 重構建議
* 技術決策
* 文件產出
* 最佳實踐

並整合你之前的高階模組（Depth / Decision / Stress Test / Self-check）。

# 🧠 程式設計與專案專用 Prompt 模板 v1.0

可直接複製使用 👇

---

```
你是一位【資深程式設計師 + 架構師 + 專案經理】。

------------------------------------
【Task 任務】

請完成以下任務：

□ 專案解析
□ 架構設計
□ 程式碼說明
□ 重構建議
□ 技術選型分析
□ 效能優化建議
□ 文件產出（README / API 文件）
□ 其他：

請具體說明要完成什麼：

------------------------------------
【Project Context 專案情境】

專案目標：
（這個專案要解決什麼問題？）

技術環境：
（語言 / 框架 / DB / Docker / 雲端環境）

專案規模：
（個人專案 / 團隊開發 / 企業級）

限制條件：
（時間 / 效能 / 安全性 / 相容性）

資料：
<<<
貼上程式碼 / 專案結構 / 需求說明
>>>

------------------------------------
【Format 輸出格式】

請使用以下格式輸出：

□ 條列式分析
□ 表格整理
□ Markdown 文件
□ README.md
□ 架構圖（文字描述）
□ 流程圖（步驟說明）
□ 重構前後對照
□ 其他：

------------------------------------
【Depth 深度層級】

請使用以下分析層級：

□ 程式邏輯層
□ 架構層
□ 系統設計層
□ 第一性原理拆解
□ 底層機制分析
□ 可擴展性分析
□ 技術債評估

------------------------------------
【Decision 決策輸出】

請提供：

1. 可行方案（至少 2–3 種）
2. 優缺點比較
3. 風險評估
4. 推薦方案與理由
5. 實作優先順序

------------------------------------
【Stress Test 壓力測試】

請額外說明：

- 若流量成長 10 倍會發生什麼問題？
- 哪裡可能成為效能瓶頸？
- 哪些地方會造成技術債？
- 哪些假設可能錯誤？
- 最壞情境是什麼？

------------------------------------
【Self-check 自我檢查】

請最後檢查：

1. 是否存在架構風險？
2. 是否有安全漏洞？
3. 是否違反 SOLID 或設計原則？
4. 是否可維護性不足？
5. 是否有更簡潔的設計方式？
```

# 🔥 進階：三種常用子模板

---

## 1️⃣ 專案快速掃描模板

```
請瀏覽整個專案並回答：

1. 專案是做什麼的？
2. 主要技術棧為何？
3. 程式進入點在哪？
4. 架構是否合理？
5. 主要風險在哪？
6. 可立即優化的 3 件事？
```

---

## 2️⃣ 架構設計決策模板

```
請設計一個可擴展架構來解決：

需求：
限制：
流量預估：

請提供：
- 架構圖（文字描述）
- 技術選型理由
- 擴展策略
- 潛在風險
```

---

## 3️⃣ 程式碼重構模板

```
請分析這段程式碼：

1. 功能說明
2. 潛在問題
3. 設計違反原則
4. 重構版本
5. 重構理由
```

* PM 視角
* 風險視角
* 可擴展視角

不只是在寫 code。

是正在設計系統。
